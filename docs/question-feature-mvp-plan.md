# 質問機能 MVP整理メモ

## このドキュメントの目的

社内で聞きづらいことを匿名で質問でき、社員同士で回答し合える「質問機能」のMVP範囲を整理するためのメモです。

現時点では、まだデザインや詳細要件が固まりきっていないため、最初から多機能にせず、まずは質問・回答の基本体験を検証できる範囲に絞ります。

---

## 全体方針

質問機能の最初の目的は、社内の人が心理的安全性を保ちながら質問できる場所を作ることです。

そのため、MVPでは「匿名で質問できる」「誰でも回答できる」「未回答や解決済みを探せる」「不適切な投稿を管理できる」ことを優先します。

一方で、カテゴリ、タグ、いいね、人気表示などは便利な機能ではありますが、最初の価値検証には必須ではないため、後から追加する方針にします。

---

## MVPで扱う機能

| 機能 | 内容 | 判断理由 |
| --- | --- | --- |
| 質問投稿 | ユーザーがタイトルと本文を入力して質問を投稿できる | 質問機能の中心となるため、最初から必要 |
| 質問一覧 | 投稿された質問を一覧で確認できる | 質問を探す入口として必要 |
| 質問詳細 | 質問本文と回答一覧を確認できる | 回答や解決状況を確認するために必要 |
| 回答投稿 | 質問に対して社員が回答できる | 質問機能の価値は回答が付くことで生まれるため必要 |
| 全社員が回答可能 | 管理者や特定メンバーに限定せず、誰でも回答できる | Yahoo知恵袋のように、知っている人が気軽に答えられる状態にしたい |
| 匿名キャラ表示 | 質問者・回答者の実名や実アイコンではなく、ランダムな名前とサムネイルを表示する | 聞きづらい質問を投稿しやすくするため、匿名性を重視する |
| 管理者による実ユーザー確認 | 一般ユーザーには匿名表示、管理者は必要時のみ実ユーザーを確認できる | 完全匿名にすると不適切投稿への対応が難しいため、運用上の安全性を残す |
| 質問ステータス | 未回答、回答あり、解決済みの状態を持つ | 質問の状況がひと目で分かるようにするため |
| 解決済みにする機能 | 質問者が質問を解決済みにできる | ベストアンサーまでは不要だが、解決したかどうかは分かるようにしたい |
| キーワード検索 | 質問タイトルや本文から検索できる | 社内ナレッジとして後から探せることが重要なため |
| 未回答フィルター | まだ回答がない質問だけを絞り込める | 回答されていない質問を見つけやすくするため |
| 解決済みフィルター | 解決済みの質問だけを絞り込める | 参考になる過去質問を探しやすくするため |
| 通報 | 質問や回答を通報できる | 匿名機能があるため、不適切な投稿に対応できる導線が必要 |
| 管理者による非表示 | 管理者が不適切な質問や回答を非表示にできる | 削除よりも運用しやすく、履歴を残したまま対応できるため |

### MVP判断の補足

MVPでは、質問機能として最低限成立する体験に絞ります。

特に重要なのは、匿名性と検索性です。匿名で質問できるだけではなく、過去の質問が社内ナレッジとして再利用できる状態にすることで、単なる掲示板ではなく、社内の質問・回答資産として活用できます。

また、匿名機能を入れる以上、通報や管理者非表示は最初から必要です。心理的安全性を高める一方で、運用上の安全性も担保する必要があります。

## 後から追加する機能

| 機能 | 内容 | 後回しにする理由 |
| --- | --- | --- |
| カテゴリ | 質問を「技術」「社内ルール」「ツール」などに分類する | 便利だが、デザインや運用が固まる前に入れると分類に迷いやすいため |
| タグ | 質問に複数のキーワードを付けて整理する | カテゴリとの違いが曖昧になりやすく、後から追加しやすいため |
| 参考になったボタン | 回答や質問に対して「参考になった」を押せる | 質問と回答の基本体験には必須ではなく、評価軸の設計が必要なため |
| いいねボタン | 投稿に対してリアクションできる | 便利だが、MVPでは回答が付くこと自体を優先したいため |
| 人気の回答 | 反応数や閲覧数などをもとに人気回答を表示する | 「人気」の定義を先に決める必要があり、初期段階では不要なため |
| ベストアンサー | 質問者が特定の回答をベストアンサーとして選ぶ | 最初は解決済みだけ分かれば十分で、後から追加可能なため |

### 後から追加する機能の判断補足

後から追加する機能は、あると便利ではあるものの、MVPの価値検証には必須ではないものです。

特にカテゴリやタグは、質問数が増えてきた段階で効果を発揮します。最初から入れると、投稿時に選択の手間が増えたり、「その他」ばかりになる可能性があります。

そのため、まずは検索と未回答・解決済みフィルターで運用し、質問が増えてから分類の必要性を判断する方が自然です。

ベストアンサーについても、将来的には有効な機能ですが、最初は「解決済み」だけで十分です。回答を1つ選ぶ体験が本当に必要かどうかは、実際の利用状況を見てから判断します。

## アイデアボックス

| アイデア | 内容 | 現時点での扱い |
| --- | --- | --- |
| 通知設定 | 回答が付いたときや質問が更新されたときの通知を細かく設定する | 将来的に利用頻度が上がってから検討する |
| メンション | 特定のユーザーに質問や回答内で言及する | 匿名性との相性を慎重に考える必要があるため、今は検討外 |
| ランキング | 回答数や貢献度をもとにユーザーをランキング表示する | 評価・称号系の機能に近いため、質問機能のMVPには含めない |
| 部署限定公開 | 特定部署だけが見られる質問を作る | 権限設計が複雑になるため、最初は全社公開を前提にする |
| 回答者指名 | 特定の人に回答を依頼する | 匿名性や心理的安全性とのバランスを検討する必要がある |
| AI回答 | AIが質問に自動回答する | 便利だが、まずは社員同士の質問・回答体験を優先する |

### アイデアボックスの判断補足

アイデアボックスに入れた機能は、将来的に検討する価値はありますが、今すぐ要件として扱うとMVPの範囲が広がりすぎるものです。

特に通知、部署限定公開、回答者指名は、権限や通知設計が絡むため、最初から入れると実装と運用が重くなります。

また、AI回答やランキングは魅力的な機能ですが、質問機能の本来の目的である「社内の人に聞ける場所」を検証した後に検討する方が良いです。

---

## 現時点でのおすすめ方針

まずは、カテゴリやリアクション機能を入れずに、質問・回答・匿名表示・検索・ステータス管理・通報管理に絞ってMVPを作るのが良さそうです。

この範囲であれば、デザインがまだ固まっていない段階でもプロトタイプとして検証しやすく、後からカテゴリやベストアンサーなどを追加する余地も残せます。

最初のゴールは、多機能な質問サービスを作ることではなく、社員が実際に「ここなら質問しても良さそう」と感じるかどうかを確認することです。
