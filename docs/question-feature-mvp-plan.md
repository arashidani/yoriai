# 質問機能：MVP整理メモ

## この資料の目的

質問機能で最初に検証する価値を整理する資料。要件確定資料ではなく、次回MTGでMVPの範囲が適切かを相談するための仮説とする。

## 現在の確度

- MVPの目的と対象範囲は仮説。
- 画面、入力制約、公開範囲、運用ルールは未確定。
- 設計方針の全体像は[設計方針サマリー](./question-feature-design-summary.md)を参照する。

## MVPで検証したい価値

社員が「ここなら質問してもよい」と感じ、回答を得られること。過去の質問も検索できる状態にして、社内の質問・回答を再利用できるようにする。

## 今回の対象

### 質問を投稿する

| 項目 | 内容 |
| --- | --- |
| 今回の扱い | MVPに含める |
| 内容 | タイトルと本文を入力して質問スレッドを作成する |
| 理由 | 質問機能の中心となる体験のため |

### 質問を探して読む

| 項目 | 内容 |
| --- | --- |
| 今回の扱い | MVPに含める |
| 内容 | 質問一覧、質問詳細、未回答・解決済みの絞り込みを提供する |
| 理由 | 質問への入口と、回答状況の確認に必要なため |

### 回答を投稿する

| 項目 | 内容 |
| --- | --- |
| 今回の扱い | MVPに含める |
| 内容 | 全社員が質問へ回答できる |
| 理由 | 知っている人が気軽に答えられることが価値の中心となるため |

### 匿名で表示する

| 項目 | 内容 |
| --- | --- |
| 今回の扱い | MVPに含める |
| 内容 | 一般ユーザーには実名・実アイコンではなく匿名キャラを表示する |
| 理由 | 聞きづらい内容でも投稿しやすくするため |

### 管理者が実ユーザーを確認する

| 項目 | 内容 |
| --- | --- |
| 今回の扱い | MVPに含める |
| 内容 | 管理者だけが必要時に投稿者の実ユーザーを確認できる |
| 理由 | 匿名性を保ちつつ不適切な投稿に対応するため |

### 質問の状態を管理する

| 項目 | 内容 |
| --- | --- |
| 今回の扱い | MVPに含める |
| 内容 | 未回答、回答あり、解決済みを表示し、質問者が解決済みにできる |
| 理由 | 質問の状況が分かり、回答受付を終える意思も表せるため |

### 質問を検索する

| 項目 | 内容 |
| --- | --- |
| 今回の扱い | MVPに含める |
| 内容 | 質問タイトル・質問本文をキーワード検索する |
| 理由 | 過去の質問を社内ナレッジとして再利用するため |

### 不適切な投稿へ対応する

| 項目 | 内容 |
| --- | --- |
| 今回の扱い | MVPに含める |
| 内容 | 質問・回答の通報と、管理者による非表示を提供する |
| 理由 | 匿名機能を安全に運用するため |

## 後から検討するもの

### 分類・評価

カテゴリ、タグ、参考になったボタン、いいね、人気表示、ベストアンサーは後から検討する。投稿数や利用者の行動を見ずに入れると、分類・評価の基準を決めにくいため。

### 利用を促す機能

通知、メンション、ランキング、回答者指名、AI回答は後から検討する。匿名性との相性や、通知・評価の運用ルールが必要になるため。

### 公開範囲を変える機能

部署限定公開は後から検討する。権限設計が増えるため、最初は全社公開を前提に検証する。

## MTGで確認したいこと

1. 上記のMVP範囲は広すぎないか、不足している必須機能はないか。
2. 匿名性と管理者による本人確認のバランスは適切か。
3. 最初は全社公開・全社員回答の前提でよいか。
4. どの機能を実装前に要件として明文化すべきか。
