# 質問機能：検索専用テーブル拡張案

## この資料の目的

質問数・回答数・検索要件が増えた後に、検索専用テーブルを導入する場合の将来案を残す。MVPで実装する設計ではない。

## 現在の確度

| 項目 | 方針 |
| --- | --- |
| MVP検索 | 質問タイトル・質問本文を通常テーブルから検索する |
| 回答本文検索 | 将来検討。検索結果の見せ方を先に決める必要がある |
| 検索専用テーブル | MVPでは作成しない |

## 検索専用テーブルを使う理由

通常の登録・表示用テーブルとは別に、質問スレッド単位で検索用データをまとめる。質問タイトル、質問本文、回答本文をまとめて検索・並び替えしやすくなる。

## テーブル案：`question_search_documents`

| 項目 | 内容 |
| --- | --- |
| 役割 | 質問スレッド単位の検索用ドキュメント |
| 主キー | `question_id` |
| 元データ | `questions`、`posts`、`answers` から作る派生データ |

### 主な項目

| カラム | 用途 |
| --- | --- |
| `title_text` | 質問タイトルを検索する |
| `question_body_text` | 質問本文を検索する |
| `answer_body_text` | 回答本文を検索する |
| `combined_text` | 複数の本文をまとめて検索する |
| `visible_answer_count` | 検索結果の表示・並び替えに使う |
| `question_status` | 状態で絞り込む |
| `last_posted_at` | 新しさで並び替える |

## 導入する場合の更新タイミング

| 操作 | 更新内容 |
| --- | --- |
| 質問投稿 | 検索ドキュメントを作成する |
| 回答投稿 | 回答本文、回答数、最終投稿日を更新する |
| 回答非表示 | 非表示回答を除外して本文・回答数を再構築する |
| 解決済み | 検索用の質問状態を更新する |

## 導入の判断

### メリット

- 質問単位で検索結果を返しやすい。
- タイトル・質問本文・回答本文ごとに検索の重み付けを調整しやすい。
- 検索時に必要な状態や件数をまとめて参照できる。

### 注意点

- 元データとの同期処理が必要になる。
- 更新漏れがあると検索結果が古くなる。
- 検索エンジンやDBの選定によって実装が変わる。

## MTGで確認したいこと

1. MVPでは質問タイトル・本文検索に限定する方針でよいか。
2. 回答本文検索が必要になる条件は何か。
3. 検索速度や検索結果の要求がどの程度になったら、専用テーブルを検討するか。
