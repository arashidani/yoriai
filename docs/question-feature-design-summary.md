# 質問機能：設計方針サマリー

> **次回MTGで相談したいこと**  
> 要求・要件がまだ未確定なため、詳細な仕様を決める前に、質問機能の方向性と設計の考え方が妥当かを確認したい。

## 1. この資料の目的

社内で聞きづらいことを匿名で質問し、社員同士で回答できる質問機能を検討している。

この資料では、現時点の設計仮説と判断理由を共有し、次の実装・詳細設計へ進んでよい方向性かを相談する。

## 3. MVPの仮説

MVPでは、**「匿名でも安心して質問でき、誰かが回答でき、必要な質問を後から探せる」** 体験の検証を優先する。

### 今回の対象

#### 質問・回答

質問の投稿、一覧・詳細の閲覧、全社員による回答投稿を扱う。

#### 匿名性と管理

一般ユーザーには匿名キャラを表示する。一方で、不適切な投稿に対応できるよう、管理者だけが実ユーザーを確認できるようにする。

#### 状態と検索

未回答・回答あり・解決済みを表示する。検索はまず質問タイトルと質問本文を対象にする。

#### 安全な運用

質問・回答の通報と、管理者による非表示を扱う。

### 今回は後回しにするもの

カテゴリ、タグ、リアクション、人気順、ベストアンサー、通知、メンション、ランキング、部署限定公開、回答者指名、AI回答は、利用状況や運用ルールが分かってから検討する。

## 4. 現時点の設計方針

### 方針1：質問・投稿本文・回答を役割ごとに分けて管理する

質問は、1件の質問スレッドと、その質問本文、複数の回答で構成される。本文そのものは質問と回答で共通するため、`posts` に置く。

| テーブル | 何を表すか | 主に持つ情報 |
| --- | --- | --- |
| `questions` | 質問スレッド | タイトル、質問者、状態、回答数、質問本文への参照 |
| `posts` | 質問本文または回答本文 | 本文、実投稿者、匿名キャラ、非表示状態、種別（質問/回答） |
| `answers` | 「この投稿は回答である」という情報 | 対象質問、回答本文を持つ投稿への参照 |

たとえば、質問を1件投稿すると `questions` を1件、質問本文の `posts` を1件作る。回答を1件投稿すると、回答本文の `posts` を1件と、その投稿を回答として扱う `answers` を1件作る。

**ねらい：** 質問と回答に共通する本文・投稿者・匿名表示・非表示状態を一か所で扱い、質問だけのタイトル・状態と、回答だけの参照を分かりやすく分ける。将来の回答本文検索や管理画面の横断表示にも拡張しやすい。

### 方針2：匿名表示は質問スレッドごとに固定する

同じ質問内では、同じユーザーを同じ匿名キャラとして表示する。別の質問では別の匿名キャラになってよい。

匿名キャラそのものは `anonymous_profiles` というマスタで管理する。実際にどのユーザーへどの匿名キャラを表示するかは、質問スレッドごとの対応表である `question_anonymous_profiles` に記録する。

| テーブル | 役割 |
| --- | --- |
| `anonymous_profiles` | 「ねこ」「いぬ」など、表示名・アイコンを持つ匿名キャラのマスタ |
| `question_anonymous_profiles` | ある質問内で「この実ユーザーには、この匿名キャラを表示する」という割り当て |
| `posts` | 投稿時に、割り当て済みの匿名キャラを参照して画面に表示する |

たとえば、ユーザーAが質問Xに初めて投稿すると、質問X・ユーザーA・匿名キャラ「ねこ」の対応を `question_anonymous_profiles` に作る。質問X内でユーザーAが追加投稿しても同じ「ねこ」を使い、別の質問Yでは別の匿名キャラを割り当てられる。

**ねらい：** 誰がどの回答をしたかをスレッド内で追える一方、質問をまたいだ個人の特定を避ける。

### 方針3：解決済みは質問者が回答受付を終える状態とする

質問者が解決済みにすると、以後の回答投稿は受け付けない。

**ねらい：** MVPではベストアンサー選定を入れず、質問が解決したかを簡潔に表す。

### 方針4：検索は通常テーブルから始める

MVPでは質問タイトル・質問本文を検索する。回答本文検索と検索専用テーブルは、質問量や検索体験の要求が明確になった後に検討する。

**ねらい：** 初期実装を抑えつつ、質問本文と回答本文がともに `posts` にあるため、後から回答本文検索にも拡張できるようにする。

## 5. 次回MTGで確認したいこと

1. 匿名質問・全社員回答・通報/非表示までをMVP候補とする範囲は適切か。
2. 質問スレッド（`questions`）、本文（`posts`）、回答としての参照（`answers`）を分ける方向性は適切か。
3. 匿名性を守りながら、管理者だけ実ユーザーを確認できる運用でよいか。
4. MVP検索を質問タイトル・本文に限定し、回答本文検索と検索専用テーブルを後回しにしてよいか。
5. 実装前に、公開範囲・匿名性のルール・通報の運用・検索結果の見せ方のうち、どこまで要件化すべきか。
